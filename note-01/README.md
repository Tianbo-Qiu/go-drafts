# Chapter 1

## 并发

**concurrent**：任务在时间段内**交替执行**，非同一时刻同时运行。

核心目标：构建健壮、可扩展的并发应用程序。

现代服务器普遍为**多核 CPU**，能够充分承接并利用 Go 语言原生的并发结构。
当下分布式云原生架构，天然适配程序的**水平扩展**需求。

<details>
<summary>单核环境下的并发本质</summary>

倘若运行环境仅存在单个处理器，并发依然成立。
操作系统本身就依靠时间片调度实现宏观并发。
当任务处于 IO、等待、阻塞状态时，切换执行其他任务，这是并发最原始的设计思想。
单核下仅为交错调度，无物理并行；宏观观感近似并行，微观串行执行。
并发的本质：更合理的任务排布策略，最大化利用空闲系统资源。
</details>

<details>
<summary>并发与并行</summary>

1. Concurrency（并发）是语言自带的结构、模型、编程范式
2. Parallelism（并行）完全依赖 CPU 硬件多核
3. 现代云服务器、机房节点默认全多核
4. 因此 Go 的并发结构，天然就能变现为硬件并行
</details>

## 设计思想
> We should focus mainly on writing correct concurrent programs, letting
> Go's runtime and hardware mechanics deal with parallelism.

开发者只需要关注**并发逻辑本身的正确性**，
真正的多核并行调度、底层线程映射、算力分配，全部交由 Go Runtime 与硬件自动处理。

<details>
<summary>Go 协程与传统操作系统线程差异</summary>

传统线程模型完全相反。
开发者需要手动感知用户态、内核态切换开销，管理线程生命周期、栈内存、线程间通信。
底层调度逻辑、资源边界往往需要开发者深度掌握，开发负担重。
线程重量大，创建、销毁、切换开销高，数量必须严格限制。
</details>

## Goroutine 模型特性
需要多少并发任务，即可创建对应数量的 goroutine，
无需手动管理底层线程、栈内存、资源配额与调度开销。

<details>
<summary>goroutine 数量边界思考</summary>

此前固有疑问：goroutine 是否也需要人为设置上限？
传统线程模型必须限制数量，否则系统资源会被快速耗尽。
Go 通过运行时轻量抽象与 GMP 调度，极大弱化了数量约束；
虽然理论上无硬性强制上限，但业务上仍需防范无节制创建导致内存堆积。
</details>

## Go 两套并发范式
Go 提供基于 **CSP** 的高层并发抽象，以 `channel` 作为通信载体。
通过通信实现任务协作，从根源上减少竞态条件、数据竞争等并发隐患，
思维模型贴近现实世界的任务协作逻辑。

<details>
<summary>CSP 与共享内存同步原语取舍</summary>

CSP 通信模型并非万能最优解。
对于简单状态读写、临界区保护等场景，共享内存式同步原语更简洁高效。
因此 Go 同时保留经典并发工具：互斥锁、条件变量、等待组等。
两种范式互补：优先用通道通信，临界区用锁同步。
</details>

<details>
<summary>底层原理学习的价值</summary>

以信号量、调度模型为例：
日常开发无需手动实现排序、信号量、锁结构，直接调用标准库即可。
但深入理解底层实现、调度机制、并发模型原理，能够拓展思维范式，
建立完整的底层认知，规避隐性坑点，提升整体编程素养。
这便是深挖底层而非仅调用 API 的意义。
</details>

## 算力扩展两大定律
### 阿姆达尔定律 Amdahl's Law
直观案例：
4 名施工者耗时 2.5 个月，综合成本高于 2 名施工者耗时 4 个月。
程序整体加速比，严格受限于内部**不可并行的串行代码瓶颈**。
即便程序中仅有极小比例的串行逻辑，多核扩展上限也会被大幅锁死，
单纯增加 CPU 核心无法无限提升运行速度。

### 古斯塔夫森定律 Gustafson's Law
对阿姆达尔定律的补充与反向视角：
待处理的问题规模并非固定，会随着可用算力、硬件资源同步扩张。
满足两个条件即可突破串行瓶颈限制：
1. 程序内串行基础逻辑体量固定，不随任务规模增长
2. 新增的所有硬件资源，都能被并发业务充分占用、持续工作

<details>
<summary>定律在云原生服务中的适配场景</summary>

绝大多数后端服务、网关、微服务、云原生程序都契合古斯塔夫森定律。
更多 CPU 核心可以承接更多请求、更大流量、更复杂业务计算。
串行调度部分占比恒定，额外算力始终有任务可执行。
即便框架抵达自身调度上限，新增资源依旧可用于扩容负载，实现持续扩展。
这也是 Go 极其适配现代云服务器架构的底层原因之一。
</details>
